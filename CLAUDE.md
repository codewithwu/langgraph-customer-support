# Python项目开发规范

> 面向人类开发者与 AI 编程助手（Claude Code）。

---

## 🤖 AI 助手行为准则

### 环境与依赖管理
- **激活环境**：执行 `python`/`uv`/`pytest` 前必须先 `source .venv/bin/activate` 安装依赖前 需要激活环境 `source .venv/bin/`
- **代码目录**：所有业务代码必须放入 `src/` 目录，禁止在根目录或 `src/` 外部编写 `.py` 文件
- **新增依赖**：必须用 `uv add <package>` 更新 `pyproject.toml`，禁止直接写入 import
- **环境同步**：`pyproject.toml` 修改后必须执行 `uv sync`

### 质量门禁
- **格式化**：每次修改后自动运行 `uv run ruff format . && uv run ruff check --fix .`
- **测试**：完成后立即运行关联测试，失败重试最多 3 次，仍失败则报告用户

### 操作限制
- **Git**：可执行 `git add`、`git commit`、`git push` 等操作，遵循安全规范（force push 需确认）
- **删除**：删除文件前必须确认

---

## 🛠 环境与工具

- **Python**：3.12+
- **包管理**：`uv`（禁用 pip）
- **开发工具**：`ruff`（lint/format）、`mypy`（类型检查）、`pytest`（测试）

## 📂 项目结构

```
project-root/
├── .venv/
├── src/                    # 严禁在根目录写业务 py 文件
│   ├── __init__.py
│   ├── main.py
│   ├── models/             # Pydantic / dataclass
│   ├── services/           # 业务逻辑
│   └── utils/              # 工具函数
├── tests/                  # 目录结构镜像 src/
├── scripts/
├── .env.example
├── pyproject.toml
└── uv.lock
```

### 导入规范
- 使用**绝对导入**：`from src.services.user import get_user`
- 修改 `__init__.py` 必须用 `__all__` 控制导出

## 📝 代码规范

### 风格
- 缩进 4 空格，行长 ≤120，文件结尾 1 空行

### 命名
| 类型 | 规范 | 示例 |
| :--- | :--- | :--- |
| 变量/函数 | `snake_case` | `get_user_name` |
| 类名 | `PascalCase` | `UserService` |
| 常量 | `UPPER_SNAKE_CASE` | `MAX_RETRY_COUNT` |
| 私有 | `_` 前缀 | `_internal_cache` |

### 函数设计
- 长度控制在 50 行内
- 参数优先用关键字参数
- 公共函数 100% 类型注解

### 异常处理
- **禁止**裸露 `except:` 或模糊 `except Exception:`
- 必须捕获具体异常类型并记录日志

### 安全
- 严禁硬编码密钥/Token/连接串
- 用 `os.getenv()` 或 `pydantic-settings` 读取配置
- 新增环境变量时同步更新 `.env.example`

### 文档
- 模块、类、公共方法必须写 Google 风格 Docstring

### 代码注释
- 所有代码注释使用**中文**
- 注释应清晰说明代码的功能和逻辑
- 禁止使用无意义的注释（如 `# 这是注释`）

## 🤖 AI 助手通用行为准则

> 以下准则用于减少 LLM 编程时常见的错误。偏向"谨慎"而非"速度"的权衡。

### 1. Coding 前先思考

**不要假设。不要隐藏困惑。主动呈现权衡。**

- 在实现之前：明确陈述假设，不确定时主动询问
- 如有多种解释，呈现出来而非沉默选择
- 如有更简单的方案，直接说明，有依据时主动反驳
- 如有不清楚之处，停下来，明确说出困惑点并询问

### 2. 简洁优先

**最少代码解决问题，不做任何 speculative 的设计。**

- 不添加超出需求的功能
- 不为单次使用的代码创建抽象
- 不添加未被请求的"灵活性"或"可配置性"
- 不为不可能的场景添加错误处理
- 如果写了 200 行而可以用 50 行解决，重写

自问："高级工程师会觉得这太复杂吗？"如果是的，就简化。

### 3. 精准修改

**只改必须改的，只清理自己造成的混乱。**

编辑现有代码时：
- 不"优化"相邻的代码、注释或格式
- 不重构没有坏的部分
- 匹配现有风格，即使你可能有不同做法
- 如发现无关的死代码，指出而非删除

当你的修改产生孤儿代码时：
- 移除因你的修改而变得未使用的 imports/变量/函数
- 不要移除预先存在的死代码，除非被要求

检验标准：每一行修改都应该直接追溯到用户的需求。

### 4. 目标驱动执行

**定义成功标准，循环验证直到完成。**

将任务转化为可验证的目标：
- "添加验证" → "为无效输入写测试，然后让测试通过"
- "修复 bug" → "写一个复现 bug 的测试，然后让测试通过"
- "重构 X" → "确保重构前后测试都通过"

对于多步骤任务，陈述简要计划：
```
1. [步骤] → 验证: [检查点]
2. [步骤] → 验证: [检查点]
3. [步骤] → 验证: [检查点]
```

强的成功标准让你能独立循环；弱的成功标准（"让它能工作"）需要不断确认。

**这些准则有效时：** diff 中减少不必要的修改、减少因过度复杂而重写、澄清性问题在错误之前而非之后提出。

## 📦 Git 提交规范

**格式**：`<type>: <描述>`（中文）

**描述规范**：无特殊说明时，描述使用中文，简洁说明本次提交做了什么（不超过 50 字）

| Type | 说明 |
| :--- | :--- |
| `feat` | 新功能 |
| `fix` | 修复 Bug |
| `docs` | 文档更新 |
| `style` | 格式调整 |
| `refactor` | 重构 |
| `test` | 测试相关 |
| `chore` | 工具变动 |

**示例**：
- `feat: 增加用户登录日志记录功能`
- `fix: 修复并发状态下的数据库连接泄漏问题`

### 变更记录
- 代码提交前，必须在 `CHANGELOG.md` 中记录本次修改的内容
- 使用 `[Unreleased]` 标签记录未发布的变更
- 格式参考 `CHANGELOG.md` 文件结构

## 📎 常用命令

```bash
# 环境
source .venv/bin/activate
uv sync

# 开发
uv run python src/main.py
uv run pytest -v
uv run ruff format . && uv run ruff check --fix .
uv run mypy src/

# 依赖
uv add <package>           # 生产依赖
uv add --dev <package>    # 开发依赖
```
